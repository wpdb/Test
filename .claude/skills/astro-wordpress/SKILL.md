# Astro WordPress Development

## Rules

- Astro uses `class=` (not `className=`), frontmatter between `---` markers, no React hooks
- API endpoints: Use `.ts` files (e.g., `src/pages/api/submit.ts`) with `export const prerender = false;` at the top
- WordPress content: `set:html={post.content.rendered}`
- Images: import local images; remote images need explicit width/height
- **Tailwind V4 only** (not V3) - never use `<style>` tags or inline CSS; prefer theme tokens (`bg-surface`, `text-content`, `bg-primary`)
- Clickable elements must have `cursor-pointer`
- Flag security issues; make minimal diffs
- Frontmatter uses TypeScript - always define proper types/interfaces

## Generated Files (DO NOT MODIFY)

These files are auto-generated and may be overwritten. **Never edit them:**
- `.phantomwp/runtime/**` - Managed PhantomWP framework code (the WordPress client lives here as of infra 1.25). Reachable as `@phantomwp/wordpress`.
- `src/lib/wordpress-config.ts` - your WordPress connection (URL + secret + image mode), rewritten on WP reconnect.
- `src/lib/wordpress.ts` - thin shim that re-exports `@phantomwp/wordpress` so legacy `../lib/wordpress` imports keep resolving. Don't edit.
- `src/lib/navigation.ts` - navigation utilities
- `src/components/menus/*.astro` - regenerated by Menu Builder
- `src/layouts/BaseLayout.astro` - base HTML shell

**Instead, use these extension points:**
- `src/lib/functions.ts` - custom data-fetching helpers (import `makeCMSRequest` from `@phantomwp/wordpress`)
- Menu slots: `<MainMenu><a href="/link">Custom</a></MainMenu>` (default slot = desktop, `mobile` slot = mobile menu)
- Create new components in `src/components/`

## Dev Server

Astro HMR handles .astro, .css, and .ts changes automatically. Only restart the dev server after installing packages or changing config files (astro.config.mjs, package.json, tsconfig.json).

## Rendering

SSG default, SSR opt-in.

- Every `.astro` page should start with `export const prerender = true;` unless the user explicitly needs per-request SSR.
- API endpoints under `src/pages/api/*.ts` must use `export const prerender = false;`.
- Static pages can still fetch WordPress data at build time. Use this for marketing pages, blogs, product listings, and normal content pages.
- SSR is only for request-specific output: private account pages, auth-gated content, previews that require request cookies, or other per-user state.
- DO NOT silently flip a page to SSR just because client-side auth state is involved. Prefer client-side hydration for login/account widgets unless the user explicitly requests server-rendered private content.

AUTH LIMITATION: generated WordPress auth stores the JWT in `localStorage`, not an httpOnly session cookie. Server code cannot read that token with `Astro.cookies.get('wp_auth_token')`, and you must not invent that cookie. If true SSR per-user rendering is required, first confirm adding a session-cookie mirror such as a new `/api/auth/session` route, then update the auth flow deliberately.

## Discovering WordPress Data

Before building pages, understand what data is available.

When PhantomWP tools are available, use:

1. **get_wordpress_schema** - Returns the full site schema: all post types (with rest_base), taxonomies, ACF field groups and fields, and detected plugins. Always call this first.
2. **fetch_wp_sample** - Fetches a real WordPress post with actual ACF data values. Use when `get_wordpress_schema` shows "unknown" field types or you need to see repeater/flexible content sub-field structures. Pass `restBase` (e.g., "team-member") and optionally `id` for a specific post.

If PhantomWP tools are unavailable, inspect `@phantomwp/wordpress` (the framework at `.phantomwp/runtime/lib/wordpress.ts`), `src/lib/wordpress-config.ts` (your connection info), `src/lib/functions.ts`, existing pages, `docs/ai-instructions.md`, and `docs/woocommerce.md` if present.

For manual queries in code, use `makeCMSRequest` (auth is automatic -- secret comes from `src/lib/wordpress-config.ts`):

```typescript
// makeCMSRequest returns { success, data, error } -- always destructure data
const { data: types } = await makeCMSRequest('/wp/v2/types');         // post types
const { data: taxonomies } = await makeCMSRequest('/wp/v2/taxonomies'); // taxonomies
const { data: sample } = await makeCMSRequest('/wp/v2/{rest_base}?per_page=1&_embed&acf_format=standard'); // inspect structure (acf_format=standard expands ACF values)
```

Custom post types have their own REST endpoint (`rest_base`). ACF fields appear in the `acf` property. See [wordpress.md](wordpress.md) for full details.

## WordPress Functions (@phantomwp/wordpress)

```typescript
getPosts()                              // Get all posts
getPost(slug)                           // Get single post by slug
getPages()                              // Get all pages
getPage(slug)                           // Get single page by slug
getFeaturedImageUrl(post, size)         // sizes: thumbnail, medium, large, full
formatDate(date)                        // Format a date string
makeCMSRequest(route, body?, options?)  // Custom WordPress REST API request
submitFluentForm(formId, fields)        // Submit a Fluent Form
createComment(postId, content, {name, email})
```

## Fonts

- **FIRST** check `package.json` for `@fontsource/*` - if font exists, just update theme.css (no install needed!)
- To install new: `npm install @fontsource/{id}` then import in `BaseLayout.astro`
- To use: update `--font-sans` in `src/styles/theme.css`
- `@fontsource/inter` and `@fontsource-variable/inter` both use font-family: `'Inter'`

For detailed font guide, see [fonts.md](fonts.md)

## Tailwind V4 Theme (src/styles/theme.css)

Tokens in `@theme { }` blocks create Tailwind utility classes. Prefer these tokens over hard-coded values in EVERY new component. PhantomWP's Theme Studio writes this file; reach for the existing token before inventing a literal.

**Color** (drives `bg-*`, `text-*`, `border-*`):
- Brand: `--color-primary`, `--color-secondary`, `--color-accent` (each with `-dark` / `-light` variants)
- Surface: `--color-surface` (page bg), `--color-surface-alt` (cards), `--color-surface-hover`
- Content: `--color-content`, `--color-content-light`, `--color-content-lighter`
- Outline: `--color-outline`, `--color-outline-light`
- Status: `--color-success`, `--color-warning`, `--color-error`

**Typography** (`--font-*` and `--text-*` utilities):
- Families: `--font-sans`, `--font-heading`, `--font-mono`
- Sizes: `--text-xs` ... `--text-4xl` (use `text-base`, `text-lg`, etc.)
- Line height: `--leading-tight` / `-snug` / `-normal` / `-relaxed`
- Letter spacing: `--tracking-tight` / `-normal` / `-wide`
- Weight: `--font-weight-regular` / `-medium` / `-semibold` / `-bold`

**Spacing**:
- `--spacing` — Tailwind v4 master unit. Every `p-*`, `m-*`, `gap-*`, `space-*` utility is `calc(var(--spacing) * N)`. Change this single token to re-scale the whole site.
- `--space-xs` ... `--space-3xl` — named vocabulary for hand-written CSS (`padding: var(--space-md)`).

**Radius / shadow / border**:
- `--radius-sm` ... `--radius-xl`, `--radius-full` (drives `rounded-*`)
- `--shadow-sm` ... `--shadow-xl`, `--shadow-inner` (drives `shadow-*`)
- `--border-hairline` / `-thin` / `-thick` (use in custom CSS `border-width`)

**Motion** (transitions, animations):
- Duration: `--duration-fast` (150ms), `--duration-normal` (250ms), `--duration-slow` (400ms)
- Easing: `--ease-in`, `--ease-out`, `--ease-in-out`, `--ease-spring`
- Pattern: `transition: all var(--duration-normal) var(--ease-out)`

**Layout primitives**:
- `--breakpoint-sm/md/lg/xl/2xl` — responsive thresholds (also wired into Tailwind's `md:`, `lg:` variants)
- `--z-base/dropdown/sticky/overlay/modal/toast` — z-index scale, prevents stacking-context wars
- `--opacity-disabled` (0.4), `--opacity-muted` (0.6), `--opacity-overlay` (0.7)

For detailed examples per category, see [tailwind.md](tailwind.md)

## Layouts

**NEVER use BaseLayout directly in pages.** BaseLayout is the low-level shell (head, scripts, styles) -- it does NOT include Header, Footer, or page structure.

Always use one of these:

- **PageLayout** (`src/layouts/PageLayout.astro`) - For pages with custom components, grids, cards. No prose styling. Use as `layout: '@layouts/PageLayout.astro'` in MDX.
- **ContentLayout** (`src/layouts/ContentLayout.astro`) - For text-heavy MDX pages (articles, docs). Includes `prose` typography. Use as `layout: '@layouts/ContentLayout.astro'` in MDX.

MDX frontmatter example: `layout: '@layouts/PageLayout.astro'`

For .astro pages, import and wrap: `import PageLayout from '@/layouts/PageLayout.astro';` then use `<PageLayout frontmatter={{ title: 'My Page' }}><slot content /></PageLayout>`

## Astro Component Patterns

```astro
---
interface Props {
    title: string;
    description?: string;
}
const { title, description = '' } = Astro.props;
---

<div class="bg-surface-alt rounded-lg p-6 border border-outline">
    <h3 class="text-lg font-semibold text-content">{title}</h3>
    {description && <p class="text-content-light mt-2">{description}</p>}
    <slot />
</div>
```

- Use `<slot />` for children (not `{children}`)
- Use `class:list={[...]}` for conditional classes
- Dynamic routes: `[...slug].astro` with `getStaticPaths()`
